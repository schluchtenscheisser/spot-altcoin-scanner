# Ticket 0.1B — Make generated code map AI-safe

## Status

Ready for review / implementation.

## Objective

Update the code-map generation workflow so `docs/code_map.md` remains an auto-generated structural repository map, but no longer misleads Codex or other AI agents into treating all listed files as equally active Independence Release architecture.

The generated code map must clearly state that it is structural navigation only, not architecture authority, and that active-vs-legacy interpretation is defined by `docs/AI_CONTEXT_CURRENT.md` and `docs/canonical/WORKFLOW_CODEX.md`.

This ticket has a small technical component: update the generator script and regenerate the generated file.

---

## Background

Phase 0 reset the AI context layer:

- `docs/AI_CONTEXT_CURRENT.md` was added.
- `docs/GPT_SNAPSHOT.md` was regenerated for the current Independence Release context.
- The old GPT snapshot was archived.
- The stale addendum was marked superseded.

Ticket 0.1A updated `docs/canonical/WORKFLOW_CODEX.md` so Codex no longer treats stale docs, legacy pipeline modules, or generated context artifacts as architecture authority.

The remaining risk is `docs/code_map.md`.

`docs/code_map.md` is generated by:

```text
scripts/update_codemap.py
```

The generator currently emits a structural map of Python modules, classes, functions, imports, call graph data and coupling statistics. This is useful, but the file is semantically blind: it may list active Independence Release modules and legacy/reference-only modules side by side.

The current generator also contains stale footer references such as:

```text
docs/spec.md
docs/dev_guide.md
reports/YYYY-MM-DD.md
```

Those references must not remain as active guidance in the generated code map.

---

## Scope

Update only:

```text
scripts/update_codemap.py
docs/code_map.md
```

`docs/code_map.md` must be changed only by regenerating it through `scripts/update_codemap.py`.

Do not manually patch generated sections of `docs/code_map.md`.

### 2. Add an AI-safety header to the generated code map

Update `generate_header()` in `scripts/update_codemap.py` so every generated `docs/code_map.md` contains a clear AI-safety / authority warning.

The header must include these points:

- `docs/code_map.md` is auto-generated structural navigation.
- It lists files, symbols, imports, function dependencies and coupling statistics.
- It is not architecture authority.
- It does not define active-vs-legacy status.
- It may list legacy/reference-only files because they still exist in the repository.
- For current AI-context routing, use `docs/AI_CONTEXT_CURRENT.md`.
- For Codex implementation workflow, use `docs/canonical/WORKFLOW_CODEX.md`.
- For current generated AI onboarding context, use `docs/GPT_SNAPSHOT.md`.
- For active-vs-legacy interpretation, do not infer from this file alone.
- Codex must not edit legacy/reference-only modules merely because they appear in the code map.

Suggested header text:

```md
## AI Safety / Authority Notice

This file is auto-generated structural navigation.

It lists repository files, symbols, imports, function dependencies and coupling statistics. It does not define architecture authority, current domain contracts, or active-vs-legacy status.

Important:
- Files listed here are not automatically active Independence Release architecture.
- Legacy/reference-only files may appear because they still exist in the repository.
- Interpret this file through `docs/AI_CONTEXT_CURRENT.md` and `docs/canonical/WORKFLOW_CODEX.md`.
- Use `docs/GPT_SNAPSHOT.md` as current AI onboarding context.
- Do not infer that `scanner/pipeline/` or other legacy modules are active target architecture merely because they appear in this map.
```

Exact wording may differ, but all semantic points above must be preserved.
